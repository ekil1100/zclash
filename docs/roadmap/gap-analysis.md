# Gap Analysis（Phase 0 草稿）

> 状态：DOING  
> 更新时间：2026-02-11 03:31 (GMT+8)

## 1. 总结

当前 zclash 的主要差距集中在：
1) 契约一致性（CLI/API/TUI 统一模型不够完整）
2) 可量化能力（benchmark 与稳定性门禁未成体系）
3) 生态兼容性（面向 mihomo/clash 的迁移路径需系统化）

## 2. 差距分级

## P0（必须优先补齐）
- CLI 术语、命令语义、错误输出不统一
- API v1 资源模型与错误码体系未锁定
- benchmark 场景与指标体系刚建立，缺实测数据回填

## P1（第二优先级）
- TUI 信息架构可用但不够“首屏可读 + 低学习成本”
- 规则与诊断链路边界场景覆盖不足

## P2（可并行推进）
- 兼容层与迁移工具
- 面向生态的文档与样例沉淀

## 3. 风险与依赖

- 依赖：需明确当前代码实现与现网可跑样例（用于实测）
- 风险：若先做功能扩展而不先锁契约，后续返工概率高
- 风险：性能优化若无统一场景与口径，结论不可复现

## 4. Phase 1 入口条件（草稿）

满足以下条件后进入 Phase 1：
1. `baseline.md` 能力矩阵完成并校验
2. `scenarios.md` 与 `metrics.md` 具备可执行口径
3. 差距优先级（P0/P1/P2）达成共识
4. Phase 1 范围冻结（CLI 先行）
